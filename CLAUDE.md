# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & Run

This is a single-module Maven project (Java 17, Spring Boot 3.4.4 parent).

```
mvnw.cmd clean install        # build (Windows)
mvnw.cmd test                 # run tests (there are currently no meaningful tests — see below)
```

Run a single test class/method the normal Maven way once tests exist, e.g.:
```
mvnw.cmd test -Dtest=ClassName#methodName
```

There are two unrelated `main` entry points in this codebase (see Architecture below) — `mvn spring-boot:run` will only work for the Spring Boot pipeline app, since the Freemarker/iText demo is a plain `main()` with no `@SpringBootApplication`. To run a specific class, use your IDE or:
```
mvnw.cmd spring-boot:run -Dspring-boot.run.main-class=com.anand.pdf.PdfProcessorApplication
```

### Ghostscript dependency

`CompressionService` shells out to a hardcoded Ghostscript binary path (`C:\Program Files\gs\gs10.07.1\bin\gswin64c.exe`). Ghostscript must be installed at that path on Windows for the PDF pipeline to run end-to-end. If a run hangs, find/kill the stray `gswin64c.exe` process (see HELP.md):
```
tasklist | findstr gs
taskkill /PID <pid> /F
```

## Architecture

The repo actually contains **two independent, unrelated proof-of-concept apps** under `src/main/java`. Don't assume code in one package is wired up with the other.

### 1. `com.anand.pdf` — Spring Boot async PDF request pipeline (the "real" app)

Entry point: `PdfProcessorApplication` (`@SpringBootApplication`, `@EnableScheduling`). This simulates a backend that picks up "PDF requests," merges/stamps/compresses their source PDFs, and writes a final PDF per request. Flow:

1. `RequestPoller` (`@PostConstruct`) loads jobs from `MockRequestRepository` (a hardcoded stand-in for a `PDF_REQUEST` DB table — see comment in that file for the intended SQL) and pushes `RequestJob`s onto `RequestQueue`, an in-memory `BlockingQueue`.
2. `PdfWorker` (`@PostConstruct`) spins up 4 raw `Thread`s (not the `pdfExecutor` `ExecutorService` bean from `ExecutorConfig` — that bean exists but isn't actually used anywhere yet) that loop forever taking jobs off `RequestQueue` and calling `RequestProcessor.process(job)`.
3. `RequestProcessor` runs each job through, in order:
   - `BlobStorageService.loadDocuments(requestId)` — lists `*.pdf` files from `src/main/resources/mock-data/request-<id>/` (stand-in for blob storage).
   - `PdfMergeService.merge(files)` — merges via PDFBox `PDFMergerUtility`, using `MemoryUsageSetting.setupTempFileOnly()` to avoid loading large merges into heap.
   - `FooterService.applyFooter(merged, "CONFIDENTIAL")` — stamps a footer onto every page via a raw PDFBox content stream.
   - `CompressionService.compress(footerPdf)` — shells out to Ghostscript to recompress (`/ebook` settings), then re-opens the result with PDFBox to validate it isn't corrupt.
   - Final PDF is copied to `output/request-<id>.pdf`.
4. `MemoryUtil.print(stage)` is a debug helper called before/after the footer+merge step; it logs heap usage and prints the on-disk size of `mock-data/request-1001` specifically (hardcoded request id — won't reflect other requests).

Mock test data under `src/main/resources/mock-data/request-1001/` and `request-1002/` is generated by `LargePdfGeneratorUtility.main()`, which calls `LargePdfGenerator.createLargeTextPdf(100, ...)` 20 times to produce large multi-page PDFs filled with random UUID text — this is meant to simulate realistic large-document load for testing the merge/compress pipeline's memory behavior. `CreatePdf.createPdf(...)` is a smaller, simpler PDFBox helper for generating trivial single-page PDFs (see commented-out sample calls at the bottom of that file).

`OutOfMemoryError` is caught per-job inside the `PdfWorker` thread loop and just logged + triggers `System.gc()` — a job that OOMs does not get retried or marked failed (see the commented-out `markRequestFailed` call), so this pipeline is still a work-in-progress around failure handling.

### 2. `com.anand.freemarkertext` — standalone Freemarker + iText HTML-to-PDF demo

`FreemarkertextApplication` has `@SpringBootApplication` commented out, so it is not a working Spring Boot entry point despite the class name — it's dead/unused. The actual working demo is `PdfGenerator.main()`, which: loads `src/main/resources/templates/template.ftlh` via plain Freemarker (not Spring's templating integration), renders it to an HTML string with a data model (e.g. `gender`, `subscribe`), then converts that HTML to `output.pdf` (repo root) using iText `html2pdf`. This package has no dependency on the `com.anand.pdf` pipeline or vice versa.
